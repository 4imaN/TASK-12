# RailOps Offline Schedule & Inventory Control Suite

A self-contained, offline-first web application for regional rail operators to plan trips, manage seats and fares, and track onboard supplies with full auditability — no internet required.

## Architecture

| Layer     | Technology            | Purpose                            |
|-----------|-----------------------|------------------------------------|
| Frontend  | Vue 3 + Pinia + Router| Role-based SPA                     |
| Backend   | Koa (Node 18+)        | REST API (HTTPS required; HTTP only in test mode) |
| Database  | MySQL 8.0             | System of record, audit trails     |
| Runtime   | Docker Compose        | Full-stack orchestration           |

All services communicate over a local bridge network. Both the backend (port 3443) and frontend (port 8443) serve HTTPS using TLS certificates from `backend/certs/`. TLS is mandatory in all operational modes; HTTP is only permitted when `NODE_ENV=test` or `SECURITY_MODE=test` (isolated test runs). The frontend dev server (Vite) proxies `/api` requests to the backend over HTTPS.

## Roles

| Role               | Access                                                       |
|--------------------|--------------------------------------------------------------|
| Guest              | Read-only trip search on published schedules                 |
| Host               | Schedule management, inventory workflows (station-scoped)    |
| Platform Operations| Approvals, audit, user admin, backups, cross-site access     |

## Prerequisites

- Docker & Docker Compose
- Node.js 18+ (for local development without Docker)
- OpenSSL (for TLS certificate generation)

## Quick Start

### 1. Bootstrap the environment

```bash
npm run bootstrap
# or: ./scripts/bootstrap.sh
```

This creates `.env` from `.env.example`, generates random secrets for `PHONE_ENCRYPTION_KEY` and database passwords, and generates self-signed TLS certificates into `backend/certs/`. Re-run with `--force` to regenerate everything.

### 2. Start with Docker Compose

```bash
docker-compose up --build
```

This starts MySQL, the backend API (HTTPS at https://localhost:3443), and the frontend (HTTPS at https://localhost:8443). Both backend and frontend serve over HTTPS using the certificates in `backend/certs/`.

### 3. Access the application

- **Frontend**: https://localhost:8443
- **API Health**: https://localhost:3443/api/health

### Manual Setup (without Docker)

> **Security note:** TLS is mandatory in all modes except `NODE_ENV=test`. HTTP without TLS is only allowed when `SECURITY_MODE=test` (set automatically when `NODE_ENV=test`). For production LAN deployment, always use proper TLS certificates.

```bash
# 0. Bootstrap (creates .env, secrets, and TLS certs in backend/certs/)
npm run bootstrap

# 1. Start MySQL (ensure MySQL 8 is running on localhost:3306)

# 2. Backend
cd backend
npm install
export DB_HOST=localhost DB_USER=root DB_PASSWORD="" DB_NAME=railops
export TLS_CERT_PATH=./certs/server.crt TLS_KEY_PATH=./certs/server.key
node src/database/migrate.js
node src/database/seed.js
npm run dev   # Starts HTTPS on port 3443

# 3. Frontend (auto-detects certs from ../backend/certs/)
cd frontend
npm install
npm run dev   # Starts HTTPS on port 8443

# For test-only HTTP mode (no TLS):
# NODE_ENV=test node src/index.js
```

> The frontend Vite dev server auto-detects TLS certs from `backend/certs/` and serves HTTPS. Override with `TLS_CERT_PATH` / `TLS_KEY_PATH` env vars or `API_PROXY_TARGET` if your backend runs elsewhere.

## Environment Variables

See `.env.example` for the full list. Key variables:

| Variable           | Default              | Description                    |
|--------------------|----------------------|--------------------------------|
| MYSQL_DATABASE     | railops              | Database name                  |
| DB_HOST            | mysql                | MySQL host (`localhost` for local dev) |
| DB_PASSWORD        | (generated by bootstrap) | MySQL password              |
| TLS_CERT_PATH      | /app/certs/server.crt| Path to TLS certificate (Docker default; local dev: `./certs/server.crt` from `backend/`) |
| TLS_KEY_PATH       | /app/certs/server.key| Path to TLS private key        |
| PHONE_ENCRYPTION_KEY | (generated by bootstrap) | AES-256 encryption key for phone numbers (required, min 16 chars) |
| BACKUP_PATH        | /backups             | Backup target directory        |

> **Note:** Sessions use random tokens stored as SHA-256 hashes in the database. No shared `SESSION_SECRET` is needed. The `.env` file, TLS private keys, and certificates are **never committed** to version control — the `backend/certs/` directory ships empty (`.gitkeep` only). Run `npm run bootstrap` to generate all secrets and TLS certificates locally before first use.

## Performance Verification

The spec requires cached hot searches to return in under 200ms. Run the benchmark:

```bash
API_BASE_URL=https://localhost:3443 ./scripts/benchmark-search.sh
```

**Pass criteria**: Warm (repeated) search runs complete in under 200ms on local network hardware. The script measures cold and warm response times for trip search and hot-search endpoints. The benchmark script exits non-zero if warm searches exceed 200ms, making it suitable for CI gating.

## Testing

Run all tests at once (includes preflight backend health check for API tests):

```bash
./run_tests.sh
```

The test runner defaults to `API_BASE_URL=https://localhost:3443`. Override if your backend runs elsewhere:

```bash
API_BASE_URL=https://localhost:3443 ./run_tests.sh
```

Run each category individually:

```bash
# Unit tests (no backend needed)
cd unit_tests && npx jest --forceExit

# Frontend tests (no backend needed)
cd frontend && npx vitest run

# API tests (requires running backend on https://localhost:3443)
cd API_tests && API_BASE_URL=https://localhost:3443 npx jest --forceExit --runInBand
```

- **Unit tests** (`unit_tests/`): Fuzzy matching, validators, masking, auth middleware, session lifecycle, backup path validation
- **Frontend tests** (`frontend/`): Vue stores, components, router, and page-level flow tests via Vitest
- **API tests** (`API_tests/`): Auth, trips, schedules, inventory, stations, users, backups, data quality, authorization, security regression, session-cap enforcement (requires running backend)

To see exact test counts run `grep -rcE '^\s*test\(' unit_tests/ API_tests/ --include='*.test.js'`.

## HTTPS Verification Checklist

After setup, verify the secure transport chain is working:

- [ ] **TLS certs generated**: After `npm run bootstrap`, `ls backend/certs/server.crt backend/certs/server.key` succeeds. (These files are never committed — they are generated locally.)
- [ ] **Backend HTTPS**: `curl -k https://localhost:3443/api/health` returns `{"success":true,...}` with `"tls":"active"`.
- [ ] **Frontend HTTPS**: Open https://localhost:8443 in a browser (accept the self-signed cert warning). The page loads.
- [ ] **Secure cookie**: Log in via the frontend. In browser DevTools (Application > Cookies), the `railops_session` cookie has `Secure` and `HttpOnly` flags set.
- [ ] **Backup verification**: After running a backup via `POST /api/backups/run`, the backup record at `GET /api/backups` shows `status: "completed"`, a non-null `checksum` (SHA-256), and a valid `file_size`.
- [ ] **Docker Compose**: `docker-compose up --build` starts all three services and both https://localhost:3443/api/health and https://localhost:8443 respond over HTTPS.

If the frontend falls back to HTTP (no `https://` in the address bar), check that `backend/certs/` contains valid cert files. Re-run `npm run bootstrap -- --force` to regenerate.

## Key Features

- **Trip Search**: Fuzzy station matching, date suggestions, hot-search caching (<200ms)
- **Schedule Versioning**: Draft workspace, pre-publish checklist, side-by-side diff, approval flow
- **Inventory Management**: Receiving, shipping, returns, stock counts with batch/serial tracking
- **Security**: bcrypt (cost 12), device fingerprinting, recovery codes, progressive lockout
- **Audit Trail**: All important actions logged with actor, timestamp, and detail
- **Backup & Recovery**: Nightly full + 15-min incremental, quarterly restore drills
- **Data Quality**: Write-time validation, daily reports, issue tracking, point-in-time backtracking

## Runtime Verification Requirements

The following guarantees **cannot be verified through static analysis alone** and require runtime or manual validation during deployment and acceptance testing:

| Claim | How to Verify | Script / Endpoint |
|-------|--------------|-------------------|
| Backup execution completes successfully | Run a backup and confirm `status: "completed"` with a valid SHA-256 checksum | `scripts/backup.sh`, `POST /api/backups/run` |
| Restore drills recover data accurately | Execute a restore drill and verify row-count match between live and restored schemas | `scripts/restore-drill.sh`, `POST /api/backups/drill` |
| TLS is active in deployment | Confirm the backend serves HTTPS and the `/api/health` endpoint reports `"tls":"active"` | `curl -k https://localhost:3443/api/health` |
| Hot-search response time < 200ms | Run the benchmark script on target LAN hardware | `scripts/benchmark-search.sh` |
| Incremental backups run every 15 minutes | Observe the backup scheduler logs over a 30-minute window | Backend container logs |
| Daily data quality reports generate at 05:00 | Confirm report records appear in `data_quality_reports` after the scheduled time | `GET /api/data-quality/reports` |

These items are documented as designed features, but their correctness depends on the runtime environment (hardware, network, MySQL configuration, cron timing). A static audit can confirm the code paths exist but not that they execute successfully in production.
